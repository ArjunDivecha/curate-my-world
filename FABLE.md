## Assessment

**The code is more sound than a repo this age usually is.** The recent commit stream shows real
engineering discipline: deterministic timezone math (`curate-events-api/src/utils/timeZoneDate.js`,
two-pass DST-safe offset), fail-loud cache/LLM handling in the scraper, and a genuinely hardened
preview proxy with SSRF + host-allowlist protection (`curate-events-api/src/routes/preview.js`). The
architecture (cache-first, single-source-of-truth category mapping, background precompute) is the right
shape for the problem. This is not a rewrite candidate. The gaps are about *durability of the fixes* and
*honest failure*, not fundamental design.

Concrete findings, worst first:

- **Zero automated tests.** `curate-events-api/package.json:14` is `"test": "jest --passWithNoTests"`;
  there is no `tests/` directory and `npm test` prints "No tests found, exiting with code 0" (verified
  this session). Every correctness fix in the README's "June 2026" list — timezone, category fan-out,
  undated-drop, route-order — is unguarded. The git log shows these exact bug classes recurring; nothing
  stops the next refactor from silently re-introducing them. This is the top risk.

- **Silent degradation contradicts the repo's own FAIL-IS-FAIL rule.** `venueCacheDb.js` swallows
  Postgres errors and returns `null`/`false` with only `logger.warn` (readAllCategoriesCache ~375-393,
  writeAllCategoriesCache ~395-414). `events.js` does check the `false` write return and throws
  (~310-313), but that throw is then caught and only `logger.error`'d (~316-318). On the read path, a
  DB lookup failure is caught and the handler falls through to an **empty response** (~438-443) rather
  than signalling an error. End-to-end, a Postgres outage produces log lines and an empty events page —
  no alert, no non-200 — which is exactly the "silent fallback" the project bans.

- **Scheduler has no retry.** `venueRefreshScheduler.js` is a self-rescheduling `setTimeout` loop
  (`scheduleNext` ~150-166); on a failed daily spawn it logs the non-zero exit (~109-115) and simply
  waits ~24h for the next slot. One transient failure = a full day of stale data.

- **Secrets were committed and a machine-specific fallback is hardcoded.** `.env` was added to git
  history (commit `0d68f6b`); keys need rotation. The scraper also reads a hardcoded absolute fallback
  path `/Users/arjundivecha/Dropbox/AAA Backup/.env.txt` (`scripts/scrape-venues.js:88,99`;
  `scripts/bakeoff-venue-extractors.js:132`) — convenient locally, but a landmine if the repo ever runs
  elsewhere. No hardcoded keys were found in tracked `src/`/`scripts/` source.

- **Cruft creates agent confusion.** `backend/` is a full, abandoned parallel TypeScript backend
  (gitignored, `.gitignore:68`) sitting beside the live `curate-events-api/`. Root Python files
  (`user_input_processor.py`, `llm_user_processor.py`, `demo_user_input.py`, `add_venue_registry_strict.py`)
  are orphaned. `src/components/WeeklyCalendar.tsx` is imported nowhere. A fresh agent can easily edit the
  wrong backend.

Config note (not a bug, worth knowing): `config.js:124-152` hard-requires only
`ticketmasterConsumerKey`; `anthropicApiKey` is optional (warn-only), so a misconfigured deploy can boot
and silently fail scraping. CORS falls back to `https://squirtle-eta.vercel.app` in prod; rate limit is
300 req / 15 min (`config.js:31-33`).

## Prioritized improvements

### P0 — Regression test suite locking the correctness invariants that keep breaking
- **What & why:** Add a real jest suite covering the three behaviors that have each broken before and
  are currently unguarded (timezone conversion, category fan-out, undated-drop), and make `npm test`
  stop being a no-op. Evidence: `package.json:14`; README "June 2026 Correctness & Safety Fixes";
  recurring bug classes across the 2026 git log.
- **Falsifiable bet:** *if* those three behaviors are each exercised by a real test, *then* `npm test`
  passes on HEAD and reverting any one fix makes it fail, *observable* via `cd curate-events-api &&
  npm test` reporting ≥1 test and exit 0, and a mutation flipping the matching test red.
- **Invariants:** `npm test` runs ≥1 real test (INV1); TM date → correct PT instant under `TZ=UTC`
  (INV2); empty-category event lands in exactly one bucket (INV3); undated TM event is dropped (INV4);
  diff stays additive and in-scope (INV5).
- **Gate intents:** `cd curate-events-api && npm test` exits 0 with ≥1 test; `TZ=UTC npm test` keeps the
  timezone assertion green; the category/undated tests assert single-bucket + drop; `git diff --name-only`
  is a subset of scope.
- **Effort:** ~half a day for a coding model. Main friction: some target logic lives inside route
  handlers in `events.js`; the contract permits *additive named exports only* so tests can import pure
  functions without rewriting the pipeline. **Full contract below.**

### P1 — Make DB/cache failures loud (align with FAIL-IS-FAIL)
- **What & why:** A Postgres outage currently degrades to an empty events page with no external signal
  (`venueCacheDb.js` ~390/412, `events.js` ~316-318 and ~438-443). This is the banned silent-fallback
  pattern. Surface it: `/api/health/deep` should report cache-write staleness/failure, and a failed
  background write should set a visible degraded flag the frontend or a monitor can read.
- **Falsifiable bet:** *if* the last all-categories write failed or the cache is older than the refresh
  interval, *then* `/api/health/deep` returns a non-healthy status for that component, *observable* by
  forcing a bad `DATABASE_URL` and asserting the deep-health payload flips to degraded/unhealthy.
- **Invariants:** a failed cache write is never reported as success; deep-health reflects true cache
  freshness; user-facing endpoints still return 200 with a clear "stale/empty — refresh failed" marker
  rather than a bare empty list.
- **Gate intents:** an integration test with an unreachable DB asserts deep-health = degraded and the
  write path logs+flags rather than returning silent success.
- **Effort:** ~half a day.

### P1 — Rotate secrets and de-hardcode the key fallback
- **What & why:** Keys are in git history (commit `0d68f6b`); the scraper hardcodes
  `/Users/arjundivecha/.../\.env.txt` (`scripts/scrape-venues.js:88,99`). Owner must rotate
  Ticketmaster/Anthropic/OpenRouter keys and update Railway + local `.env`. Code side: gate the `.env.txt`
  fallback behind an explicit env flag (default off) so the path is opt-in, not implicit.
- **Falsifiable bet:** *if* the fallback is gated, *then* the scraper fails loudly with a clear message
  when no key is present and the flag is unset, *observable* by running the scraper with no key and no
  flag and asserting a non-zero exit with an actionable error (not a silent read of the hardcoded path).
- **Invariants:** no key material printed to logs; no implicit read of an absolute machine path.
- **Gate intents:** a unit test asserts key resolution order (env → flagged fallback → hard error) and
  that the hardcoded path is not read unless the flag is set.
- **Effort:** code ~1 hour; key rotation is an **owner action** (see ARJUN.md), not a coding task.

### P2 — Bounded retry / same-day backfill for the daily scheduler
- **What & why:** `venueRefreshScheduler.js` (~109-115, ~162-165) has no retry; a transient failure
  loses a day of freshness. Add a small bounded retry with backoff before deferring to the next 6 AM slot.
- **Falsifiable bet:** *if* a scheduled run's child exits non-zero, *then* it retries up to N times with
  backoff before rescheduling for tomorrow, *observable* by injecting a failing child and asserting N+1
  spawn attempts within the window.
- **Invariants:** total retries bounded (no runaway loop); a success on retry cancels further retries.
- **Gate intents:** a unit test with a stubbed failing spawn asserts the retry count and that the next
  daily slot is still scheduled.
- **Effort:** ~2-3 hours.

### P2 — Delete the cruft (reduce agent misfire risk)
- **What & why:** The abandoned `backend/` TS tree, orphaned root Python files, and unused
  `WeeklyCalendar.tsx` invite edits to the wrong code. **Owner-gated** (global rule: no deletions
  without permission), but high-value: a coding agent that edits `backend/` instead of `curate-events-api/`
  wastes a whole session.
- **Falsifiable bet:** *if* the dead trees are removed, *then* a repo-wide import/require scan shows no
  live reference to any deleted file, *observable* via grep for imports of the removed modules returning
  empty and `npm run build` + `npm test` still green.
- **Invariants:** nothing still-referenced is deleted; frontend build and backend tests stay green.
- **Gate intents:** import-reference scan is empty for each removed path; `npm run build` and `npm test`
  exit 0.
- **Effort:** ~1 hour once the owner approves the deletion list.

## Context

Repository root: `/Users/arjundivecha/Dropbox/AAA Backup/A Working/Curate-My-World Squirtle`.
The live backend is the Node/Express package `curate-events-api/` (entry `server.js`). It
aggregates Bay Area events from the Ticketmaster API and a venue-scraper cache, normalizes
them into ~12 categories, and serves a Postgres-backed precomputed "all-categories" response
so user requests never call Ticketmaster live.

Three behaviors in this pipeline have each broken in production before and were fixed by hand
(see the git log and README "June 2026 Correctness & Safety Fixes"):
1. **Pacific-time conversion** — Ticketmaster returns venue-local (Pacific) `localDate`/`localTime`.
   `curate-events-api/src/clients/TicketmasterClient.js` converts them via
   `curate-events-api/src/utils/timeZoneDate.js` (`America/Los_Angeles`). A prior bug used the
   server's local timezone; on Railway (UTC) every event shifted 7-8 hours.
2. **Category fan-out guard** — in `curate-events-api/src/routes/events.js`, an event with an
   empty category is stamped with the category it was fetched for; without this guard one event
   appeared in every category bucket.
3. **Undated-event drop** — `TicketmasterClient.js` drops events with missing/unparseable start
   dates instead of fabricating "today".

Critically, the repo has **zero automated tests**: `curate-events-api/package.json` defines
`"test": "jest --passWithNoTests"`, and running it prints "No tests found, exiting with code 0".
Nothing prevents any of these three fixes from being silently reverted. This contract adds a real
jest regression suite that locks the three behaviors, plus a guard that `npm test` is no longer a
no-op. "Fan-out" means one event incorrectly emitted into all category buckets. "Bucket" means the
per-category array in the assembled response. All terms above use the files at the paths given.

## Build Loop vs Product Loop

The **build loop** is fully machine-checkable: the coding model resolves each gate command into a
deterministic jest invocation and a diff-scope check, then drives them to exit 0. The gates prove
that a real test suite exists, that timezone conversion is correct under a UTC process, that the
empty-category event lands in one bucket, that undated events are dropped, and that the diff stayed
inside the declared scope. Passing gates proves only that these regressions are now covered and green
on this commit in this environment.

The **product loop** is the real-world bet: that users see fewer wrong event times, fewer duplicate
events across tabs, and no phantom "today" events - measured only in production over time against user
reports and cache audits. The coding model **may not claim** the product bet is satisfied merely because
the build-loop gates pass; green gates lock the known regressions but do not prove the live product
outcome. Report gate results as build-loop evidence only.
